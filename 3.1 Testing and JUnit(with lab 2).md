
- **测试是程序员检验代码正确性的主要手段**
- 利用AI,或类似于自动评分机的程序是不符合开发实际且不够普适的
## 测试思想
### ADD（自动评分机驱动开发）
写完程序直接提交评分机，再修改编译错误，改完后，又根据报错反复修改，直到成功
- 混乱，缓慢，不安全
- 不建议使用

### Unit Test(单元测试)
将大程序分解为小程序单元，为大部分小单元编写测试集
- 较为可靠，可维护和方便处理
- 但不好测试有附属关系的测试，浪费时间
- 总体上较为推荐

### 集成测试
集成测试验证组件之间的交互是否正确
- 手动测试繁琐，自动化测试也困难。
- 在高度抽象的层面上，很容易忽略一些细微或罕见的错误
### TDD（测试开发驱动）
设计一个功能，编写测试，从零开始写程序，直到测试通过

>建议：**应该编写测试，但前提是它们可能有用**
## 测试用途
-  将复杂逻辑拆解为小单元，降低记忆负担。
- 在最小单元内快速锁定报错，避免在整个程序调用中进行盲目的全量排查。
- 方便重构，确保在优化代码性能或可读性时，不破坏既有功能的正确性


## 来自`lab2`的收获

### 对IDEA调试界面熟悉

#### 步入
- 按程序实际运行方式观察，会进入调用函数
> 在发现判断那个函数出现问题后再多次使用

#### 步过
- 不理会调用函数运行，直接给出这main函数这行代码的结果
> 用于初步判断哪个函数有问题，缩小报错函数范围

#### 步出
- 跳出当前函数，用于不小心进入深层并不想探究的函数的退出
> 一种补救措施

#### Java Visualizer
- 观察变量联系和数值变化的好插件，个人认为舒适度高于自带的`帧|变量`
> 变量，值，关系一应俱全，舒适


#### 对表达式求值
- 可以通过对某一行使用对表达式求值，改变/得知条件，用于测试问题所在得到出错处
>  有了Java Visualizer后一般个人只用于改变条件，看看条件改变后是否正确



#### 断点与条件断点
- 断点：程序运行到此处停下，允许你使用debugging界面
- 条件断点：当满足预给的表达式条件时才停下
> 条件断点能节约特别是循环到某个结点的调试时间

#### 技巧
- 不写行内嵌套函数，行中出现2个函数，难以调试（会让人选择其中一个进入，另一个跳过）
- 用中间变量承接嵌套函数，方便通过中间变量调试观察函数有无错误
- 先用 步过 找清那个函数出错，再用 步入/对表达式求值/JV 对单个函数进行细致调试

#### `Junit`相关

```java
//相关简写手法
import static org.junit.Assert.*;  
import org.junit.Test;

@Test
public void testMethod() {
     actual;
     actual =Method(actual);
     expected;
    assertEquals(expected, actual);
    assertTrue(<expected, actual);
}
//无需写主函数，添加了org.junit.@Test可以实现单元测试
```
