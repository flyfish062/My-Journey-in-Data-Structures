# 8.1 `Encapsulation,API's,ADT's`

- 关于软件工程的一些形而上学的理论


## 效率

> 在有限资源下通过精妙的抽象实现最优产出。

|          |             |                 |                                          |
| -------- | ----------- | --------------- | ---------------------------------------- |
| **编程成本** | 开发者开发与迭代的速率 | 开发周期、代码可读性、维护难度 | **核心瓶颈：** 开发者的挫败感。挫败感会显著降低编码速度，直接损害工程进展。 |
| **执行成本** | 软件对硬件资源的消耗  | 响应时间、内存占用、能耗    | 影响用户体验及基础设施运营成本。                         |

## Java自身提效特性

- **包 ：** 提供了代码组织与包级私有访问的边界，但其层级关系有时过于僵化。

- **静态类型检查：** 牺牲了一定灵活性并带来了繁琐的类型转换，但它能在编译期捕获错误，使代码易于追踪。

- **继承 ：** 实现了代码复用，但也引入了强耦合的“Is-a”关系。从调试角度看，继承会导致破碎的执行流 ，使追踪逻辑变得异常复杂

## 封装

> 模块化设计的基础


- **模块 ：** 指一组协同工作以完成特定任务或一组相关任务的方法集合。
- **封装 ：** 当模块实现细节被完全隐藏，且只能通过记录的接口进行访问时，该模块被称为是封装好的。

- 内部重构不会波及外部调用逻辑，从而确保了系统的稳定性。

### 封装作用

- **降低认知负荷**  使用者无需深入复杂的底层实现。
- **解耦：** 限制模块间的交互路径，使得各组件可以独立。
- **容错性：** 严格的访问控制防止了外部非法篡改。

## `API`

> **API（应用程序编程接口）** 是开发者之间签署的技术契约

1. **语法规范 ：**
    - 由**编译器**强制校验。
    - 确保方法签名、参数类型及返回值在调用链中物理匹配。
2. **语义规范 ：**
    - 定义了程序的“行为预期”。
    - 目前主流通过**文档**、使用示例及**单元测试用例**来定义。语义错误（即程序未按预期运行）通常比语法错误更难察觉。

## `ADT`

- 定义高层抽象逻辑，而将物理实现延迟到具体场景中

### 实现方法

1. **扩展：** `class Stack<Item> extends LinkedList<Item>`。
    - **分析：** 直接复用父类方法，但暴露了父类的非栈行为（如 `add(index, item)`），破坏了栈的严谨性。
2. **委托 ：** 内部持有一个具体的对象实例。
    - **分析：** 通过调用内部对象的方法完成任务，隐藏了不必要的父类接口。
3. **接口委托 ：** 内部持有一个 `List<Item>` 接口对象。
    - **意义**  允许在运行时注入任何实现了 `List` 接口的实例（如 `ArrayList` 或 `LinkedList`），实现了真正的多态。



## 实现方法的选择

- **拓展**:Is-a关系--当你对父类实现有深度控制，且子类确实是父类的特化版本时使用
- **委托**:has-a关系--使用某个类的功能，而不希望在身份上与其关联时使用


## 高级抽象--视图


>可以不看,也不容易懂

**视图**是对基础对象的非拷贝抽象。不产生新数据，提供一个观察和操作原数据的“窗口”。

#### 2. 底层原理：偏移映射 

- **机制**：通过内部维护一个**偏移量**，将对视图的局部索引操作动态转换为对原对象的全局索引操作。
- **公式**：view.get(k)-->original.get(k+ offset)
- **特性**：**实时反馈**。对视图的修改直接作用于原对象，保持数据强一致性。

#### **3. 三大好处**

- **降低复杂度**：消除手动计算索引（如循环起点、终点）的“无关逻辑”，规避 **差一错** 等越界风险。
    
- **逻辑复用**：实现“局部如整体”。让局部片段可以直接复用针对整体对象设计的通用函数(如 reverse(List)），提升代码整洁度。
    
- **极致效率**：**零拷贝**。无论处理多大的子集，都不涉及数据复制，空间复杂度为 O(1).

# 8.2-8.4 `Asymptotics`
> 详见算法与算法分析,这里仅补充一些知识



- 数据规模呈几何倍数增长时，执行成本的效率差异将直接决定系统的性能.
- 不关心 N=100 时的具体耗时，而只关心当 N→∞ 时算法表现出的**渐进趋势**
### 步骤

1. **仅考虑最坏情况**
2. **选定代表性操作**
3. **忽略低阶项**
4. **忽略乘法常数**

> 不仅应该学会从数学上计算估计复杂度,同时要学会画图,取简单例子估算复杂度,建立几何直觉

### `Big-Theta (Θ) 与 Big O (O) 与Big Omega (Ω)`

- **Big-Theta (Θ**)：渐进等价（紧凑界限）。精确描述了函数的上下界。如果 R(N)∈Θ(f(N))，则意味着存在正数 `k1​,k2​`，使得在 N 足够大时，`k1​⋅f(N)≤R(N)≤k2​⋅f(N)`。

- **Big O (O**)：渐进小于或等于（上界）。如果一个算法是 Θ(`N2`)，它必然也属于 O(`N2`)、甚至 O(`N3`) 。相对宽松,常用它来表达最坏情况的性能天花板。(最好分析,常用)

- **Big Omega (Ω**)：大于或等于（下界）。它保证算法至少需要这么长时间。掌握 Ω 能识别优化工作的理论终点，避免在已经达到最优的算法上浪费精力。

### 三大核心技术

- **精确求和 ：** 对核心操作进行严密的数学计数。
- **枚举实例 ：** 当逻辑复杂时，通过推演 N=1,2,3… 寻找规律。
- **绘制图形 ：** 利用几何模型（如递归树或坐标面积）来思考复杂度。

> `1+2+4+⋯+Q=2Q−1`(**2 的幂次序列求和**),学会分析嵌套循环、几何级数、递归、二分、分治 五种基本模型.
