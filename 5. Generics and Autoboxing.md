# 5.1 `Autoboxing`

> 保持高性能原始数据处理的同时，支持高度抽象的泛型数据结构

- **将Java中为追求性能的原始类型与抽象类型（对象）建立联系，让泛型接口能够以统一的对象形式处理各种数据**


# 泛型的问题

- 泛型在运行时的擦除机制，诸如 `LinkedListDeque<int>` 的声明会触发语法错误。

- 设计了包装类将原始类型与抽象类型实现统一

| 原始类型      | 包装类         |
| --------- | ----------- |
| `byte`    | `Byte`      |
| `short`   | `Short`     |
| `int`     | `Integer`   |
| `long`    | `Long`      |
| `float`   | `Float`     |
| `double`  | `Double`    |
| `boolean` | `Boolean`   |
| `char`    | `Character` |

> 通过编译检查，它们将原始数据对象化,能够参与泛型逻辑


## 自动装箱

 当程序上下文期望一个包装类（一般是泛型接口），但实际给的是原始类型时，编译器会自动注入代码将其转换为对象。



```java
ArrayDeque<Integer> dq = new ArrayDeque<>();
dq.addFirst(20); //现代版本，自动装箱
dq.addFirst(new Interger(20));//老版本，手动装箱

//这一过程等同于调用包装类的转换方法（ `valueOf` 逻辑，用以从对象中获取原始数值）。
```

> 转化为包装类，会存在64位的引用与64位的对象头开销，添加额外开销

> 自动装箱和拆箱涉及内存分配与方法调用,导致明显的性能下降，并增加垃圾回收的压力。

>**数组永远不会自动装箱或拆箱**


## 自动拆箱

- 当程序需要原始类型数值，但实际操作的是包装类对象时，编译器会自动提取其内部的原始值。

### 扩宽与收缩（原始类型间的转换）

>  基础内容

- 当程序期望接收类型 `T2`，而传入的是类型 `T1` 时，如果 `T2` 的取值范围大于 `T1`（且无精度损失风险），Java 会执行隐式转换
- 一个“宽”类型（如 `double`）转换为“窄”类型（如 `int`），由于这被视为**不安全**的操作（可能导致截断或精度丢失），Java 编译器会强制要求开发者使用显式强制转换


# 5.2 `Immutability`

> 不可变数据类型是指实例在实例化后，其实例无法以任何可观察方式发生变化的数据类型。

- Java中的string对象是不可变的----（当`String`对象被串接时，原始字符串中没有被修改——而是返回一个全新的`String`对象。）


- 任何带有非私有变量的数据类型都是可变的，除非这些变量声明为`final`
> 因为外部方法可以改变非私有变量的值，从而导致可观测的变化。


## `final`
- **变量在第一次赋值后，禁止再次更改指向或赋值**

> **final** **约束的是“引用”这个容器，而不是它所指向的“内容”**。


```java
public final ArrayDeque<String> deque = new ArrayDeque<>();
//deque变量被final修饰，不能将指向另一个新的ArrayDeque实例。
//依然可以执行 deque.add("data")。引用的地址没变，但对象内部的“内容”发生了改变。
```


## 不变性的优势

- **消除副作用**：你可以放心地将对象传递给第三方函数，而不必担心它被修改。
- **调试与测试友好**：对象的状态是恒定的，这使得单元测试变得极其简单。

> 缺点就是要创建一个新对象来更改属性

## 包装类与原始类型糅合问题


```java
@Test
public void test() { 
    ArrayMap<Integer, Integer> am = new ArrayMap<Integer, Integer>();
    am.put(2, 5);
    int expected = 5;
    assertEquals(expected, am.get(2));
}
```

> `JUnit`的`assertEquals`方法过载，例如。`assertEquals(int expected, int actual)`、`assertEquals(Object expected, Object actual)`等。因此，Java 不确定该调用哪种方法

- 将expected转换为包装类（Integer)
- 将am.get(2)转换为int


## Generic Methods


### `Implementing get()`

> 创建一个工具类`MapHelper`,创建get(`Map61B`, `key`)---解决**get 方法尝试获取 `ArrayMap` 中不存在的键，会抛出 `ArrayIndexOutOfBoundsException`**。

```java
//写出MapHelper<k.v>显然是不对的，我们要做的是让函数 拿取 静态方法所使用的 形参形式 的数据结构，而不是工具类接受泛型
public static <K,V> V get(Map61B<K,V> map, K key) {
    if map.containsKey(key) {
        return map.get(key);
    }
    return null;
}
```


> 存在功能局限：当需要对泛型对象进行比较时，普通的 `Object` 方法集显然不够用。


### `Implementing maxKey()`

- 返回给定`ArrayMap`中所有键的最大值。只有在能比较密钥的情况下才有效。


```java
//不正确
public static <K, V> K maxKey(Map61B<K, V> map) {
    List<K> keylist = map.keys();
    K largest = keylist.get(0);
    for (K k: keylist) {
        if (k > largest) {
            largest = k;
        }
    }
    return largest;
}
```

> `>` 算符不能用于比较 `K` 个物体。这只适用于原始类型

- 替换为`k.compareTo(largest)` 也不对，**对象没有实现`compareTo`方法！**

#### 修改

```java
//对静态函数实现extends Comparable<K>即可
public static <K extends Comparable<K>, V> K maxKey(Map61B<K, V> map) {...}
```

>`K extends Comparable<K>` 表示必须实现可比的接口，并且可以与其他 K 进行比较。

>写成`Comparable<K>`其实是因为`<K>`是确定是哪个数据类型要实现`CompareTo`方法

```java
public class K implements Comparable<K>{

@Override
public int compareTo(K otherK) { };
}
```


## 继承与泛型中的 extend

>  继承中的 `extends`：**扩展与赠予**。 泛型中的 `extends`：**限制与断言**。


- 虽然`Comparable<K>`是个接口，但出于在泛型中要表达的意思相通（extend本表示继承父类，有比父类更多的特性，implement表示实现对应接口的功能）
- **在泛型中都是对类型的限制：extend 表示 K 只能继承于某类型，或者 K必须实现某个功能**，核心都是限制，设计时采取了化简，统一使用extend
