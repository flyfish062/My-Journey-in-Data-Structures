# `SLList`
## 优化，调用中间人管理数据和指针

- 并基于`Intnode`重构功能函数

```java
public class IntNode {  
    /**为改善LntList数据结构存在的裸递归问题，将其重塑，删除相关方法，转变为一个中间类*/  
    public int item;  
    public IntNode next;  
    public IntNode(int x, IntNode y){  
        item =x;  
        next =y;  
    }  
}
```

[`SLList.java`](My-Journey-in-Data-Structures/DataStructures/SLList.java)

> `SLList`通过调用`IntNode`来实现对数据的添加，避免裸递归同时加强了私密性和封装性

## 一个新奇的想法--哨兵结点

- 明确执行思路：
- **实现删除/添加功能其实是找前驱结点，定位 -> 修改 next 指针**

```java
public void addLast(int x)  
{  
    IntNode p = first;  
    //处理如果建立空链表，找不到前驱结点的问题
    if(first==null)  
    {  
        first = new IntNode(x,null);  
        return;  
    }  
    //正常执行逻辑：定位 -> 修改 next 指针**
    while(p.next!=null){  
        p = p.next;  
    }  
    p.next= new IntNode(x,null);  
    size++;  
}

```

### 解决
- **创建一个不存储业务数据、仅用于简化边界处理的辅助节点(哨兵结点)作为前驱结点/后驱结点**
- **有了哨兵节点**，原本的“头节点”就变成了“第二个节点”，而哨兵节点成了永远不变的“真·头节点”。**所有节点的逻辑处理就完全统一了**

> 这时牢记sentinel.next是链表第一个元素即可正常重构整个`SLList`

[`SLList.java`](SLList2.java)

**不变式**

> 引用sentinel始终指向哨兵节点。  
最前面的项（如果存在）始终位于sentinel.next.item。  
该size变量始终表示已添加项目的总数。


# `DLList`

## 缓存再探
- 根据`SLList`的功能函数的实现方法，易看出实现`addLast`,`removeLast`需要遍历整个结构才能找到前驱结点，导致检索时间过长
### 修改思路
- 在末尾添加一个指向倒数第二个结点的指针做前驱结点实现末尾删除


> 但多次使用`removeLast`仍做不到永远有倒数第二个结点的指针做前驱结点
> **重点：缺少一种寻找前驱结点的指针...

构想：**每个元素添加一个先前的指针`IntNode`，使得指针方便移动到实现删除和添加功能的前驱结点**->`DLList`双向链表
