# overloading(过载)

- 类似于C++ 中的重载，函数名相同，返回类型/参数列表不同，引发过载
- 可以智能处理多类型参数
### 缺点
- 重复性极高，难看，源代码长度变长
- 需要维护代码增多，更改时往往更改发生过载的所有函数

- 需要一种上层类型包括不同的参数类型来获得统一性


# `Hypernyms`(上位词),`Hyponyms`(下位词),and Interface Inheritance`

- 在逻辑等级上， 一个父类型下面有多个子类型，其父类型可以称为子类型的上位词，反之为下位词
- 在Java中，这个父类被称为是子类的超类(`superclass`）

### 做法

-  步骤 1：定义通用列表上位词的类型--选择超类
- 步骤 2：指定 其子类作为该类型的下位词。

# Java与C++继承的比较

> 函数底层实现上

- Java接口只有功能函数的声明，不会有底层实现
- C++的继承一般严格继承父类成员与成员函数的底层实现

>在成员上 

- Java不能有成员变量，只能包含常量和成员函数
- C++可以包含，储存数据

> 在继承与接口上

- Java实现的是单继承，多实现
- C++是可以继承多个类
# Interface Inheritance（接口继承）的实现
 
> 接口继承指的是一种关系，其中一个子类继承了该超类的所有方法/行为
### 定义接口

```java
/** 接口：定义了列表必须具备的功能 */

public interface List61B<Item> {
    void addFirst(Item x);    
    void addLast(Item x);     
    Item getFirst();          
    int size();               
    
    // 默认方法（可选）：如果所有实现类逻辑都一样，可以用 default
    default void print() {
        for (int i = 0; i < size(); i++) {
            System.out.print("Item " + i + " ");
        }
    }
}
```

### 使用`implements`实现具体定义

```java
/** 实现类 A：基于数组的列表 (Array-based) */

public class AList<Item> implements List61B<Item> {
    private Item[] items;
    private int size;
   //必须实现接口中声明的其他所有方法
   @Override
    public void addLast(Item x) {/**具体实现*/}
    @Override
    public int size() { return size;}
    @Override
    public void addFirst(Item x) { /* 逻辑 */ }
     @Override
     public Item getFirst() { return items[0]; }
     
}
```

```java
/** 实现类 B：基于链表的列表 (Linked-list)*/
public class SLList<Item> implements List61B<Item> {
    private class Node {
        Item item;
        Node next;
    }
    private Node sentinel;
    private int size;
    @Override
    public void addLast(Item x) {/**具体逻辑*/}
    @Override
    public int size() {return size;}
    @Override
    public void addFirst(Item x) { /* 逻辑 */ }
    @Override
    public Item getFirst() { return sentinel.next.item; }  
}
```

### 使用接口

```java
public class Main {
    public static void main(String[] args) {
    //接口成员变量使用
        List61B<String> myList;
        myList = new AList<>(); 
        //或者
        myList = new SLList<>(); 

        myList.addLast("Hello");
        processList(myList); 
    }
    //接口成员函数使用
    public static void processList(List61B<String> list) {
        System.out.println("处理列表，长度为: " + list.size());
    }
}
```

### 特点


- **关键字**：使用 interface 定义接口，使用 implements 实现接口。

- **强制性**：如果一个类 implements 了接口，它**必须**重写（Override）接口中所有的抽象方法，否则无法通过编译。

-  **多态性** ：
    
    - **编译时类型 (Static Type)**：接口类型（如` List61B`）。
    - **运行时类型 (Dynamic Type)**：具体的类（如 `AList`）。
    - **效果**：程序在运行时会根据实例的真实类型去调用对应的方法。

### Overload/Override
1. **Overload (重载)**：**同类异参**。
    
    - **发生范围**：同一个类里。
    - **样子**：方法名相同，**参数必须不同**（个数或类型）。
    - **目的**：提供一个功能的多种版本。
        
2. **Override (重写)**：**父子同参**。
    
    - **发生范围**：子类与父类（或接口）之间。
    - **样子**：方法名、参数、返回值**必须完全相同**。
    - **目的**：子类要用自己的逻辑替换（覆盖）父类的逻辑。


## @Override

### 作用
- 让编译器检查方法名和参数是否写对
- 当函数名写错但在上方添加了`@Overrride`时，编译器报错提醒
- 提升代码可读性，阅读时知道是在重写函数
- 当修改了父类中的接口函数名时，提醒报错，不容易出错

> 重写父类或者接口方法，一定加`@Override`。


### 基于`GROE`关于接口继承的补充

- `GROE`规定`=`所实现的位的复制必须左右均为相同类型
```java
上位类 = new 下位类()；
//这种同样成立，下位类的类型可以融入上位类框架，GROE可以实现
```

# Implementation Inheritance

> 一种由超类定义子类函数的逻辑实现的继承思路

```java
超类
default public void print() {
    for (int i = 0; i < size(); i += 1) {
        System.out.print(get(i) + " ");
    }
    System.out.println();
}
```
则超类的所有子类都可以使用这个方法

### 有关Implementation Inheritance的要点

- 如果子类重写了超类的实现了实现继承的函数（记得加`@Overrride`)，则出现下面情况

```Java
//发生“重写”的动态方法选择
超类 name = new 子类();
name.print();
将调用子类的print实现
```
## dynamic method selection（动态方法选择）

- 在Java中一个变量可以认为有2种类型：静态类型和动态类型
- `超类 name = new 子类();`这种写法会使得name的静态类型为原定义的超类，动态类型变为子类
- `=`会将动态类型传递，故动态类型会根据当前所指对象类型而变化
- 可以理解为静态类型是编译时变量的类型，动态类型是解释时变量的实际类型
- 当 Java 运行一个**被覆盖(Override)**的方法时，它会在其动态类型中搜索合适的方法签名并执行它
- **对超载(Overload)方法不适用**
- **当发生重载时， Java检查调用哪个方法时，它会检查静态类型，并调用该方法的同类型参数。这时候动态方法选择不再适用**

## 实现继承的一些缺点
- 超类与子类的重写在某种程度上鼓励了复杂的代码
- 长继承中容易出现难以排查的实现继承，无法得知`@overrride`下的函数是单纯的重写（实现）还是子类对超类的额外重写
- 如果两个接口默认方法冲突，冲突可能难以解决
